# 轉換錯誤處理規則（M2 設計文件 3/3）

定案日：2026-07-09｜對應 spec §9 缺口「轉換錯誤處理規則」｜總原則：**嚴格全擋（F13 紅燈即不上線）**

## 內容轉換規則

| 語法 | 轉換 | 失敗時 |
|---|---|---|
| `[[筆記名]]`、`[[筆記名\|顯示字]]` | 目標筆記也是 publish → `/posts/<目標slug>/` 連結 | 目標不存在或未發佈 → **錯誤** |
| `![[圖片.ext]]` | 從 `attachments/` 複製到文章資料夾，改 `![](./圖片.ext)` | 附件找不到 → **錯誤** |
| `![[筆記名]]`（嵌入筆記） | v1 不支援 | **錯誤**（列入 backlog 逐步補） |
| Dataview / 其他外掛區塊 | v1 不支援 | **錯誤** |
| Callout `> [!note]` | v1 原樣輸出（GFM 引用呈現） | — |
| `$...$`、`$$...$$` | 原樣（網站已支援 KaTeX） | — |
| 標準 Markdown | 原樣 | — |

## 擋 CI 的錯誤（任一出現即 exit 1，全部列完才退出）

1. `slug` 缺、格式不符、或與既有文章重複
2. `published` 缺或非法日期
3. front matter schema 驗證失敗（型別錯誤）
4. wiki link 指向不存在或未發佈的筆記
5. 嵌入附件找不到
6. 不支援的語法（嵌入筆記、dataview）
7. gitleaks 掃到疑似機密

## 警告（不擋，列在 CI log）

- 缺 `description`、缺 `category`
- 單一附件 > 2MB

## CI 流程（vault repo）

```
push → gitleaks → 轉換腳本（Python）
                    ├─ 有錯誤 → 紅燈，全部錯誤一次列出，不上線
                    └─ 全過   → 產出 commit 推到網站 repo（deploy key）
                                → 網站 repo 門檻部署接手（lint→check→build→deploy）
```

- 錯誤訊息格式：`筆記路徑: 錯誤類型: 說明`，一次列出所有錯誤（不要修一個跑一次）
- 轉換腳本本身要有單元測試（對應教科書 Ch 19–20 的測試練習）
